# docs.pantavisor.io

Official documentation site for [Pantavisor](https://www.pantacor.com), built
with [Docusaurus](https://docusaurus.io/).

The site is split into **two documentation instances**:

- **Curated** (`curated/`) — hand-authored, versionless Markdown: the
  user-task information architecture, concepts, and positioning. **Pantavisor is
  PID 1 and owns the full device update** — this is the lead message and the
  curated content embodies it. Served at the site root (`/`).
- **Reference** (`reference/`) — generated, versioned, repo-shaped content
  (`meta-pantavisor`, `pantavisor`, `pvr`) ingested from each Pantavisor
  release. Served at `/reference`, with a version dropdown.

> **Rule of thumb:** author narrative and guides in `curated/` by hand. Never
> hand-edit `reference/` or `reference_versioned_docs/` — they are regenerated
> from release artifacts. Edit reference source in its originating repository.

---

## Repository layout

Only **authored** content and config are committed. The reference docs are
**downloaded and generated at build time** and are git-ignored.

```
curated/                       # Curated, versionless docs (hand-authored)  → served at /  [COMMITTED]
releases.json                  # Which reference versions to publish + which is `current`   [COMMITTED]
sidebarsCurated.ts             # Sidebar for the curated instance (user-task IA)            [COMMITTED]
sidebarsReference.ts           # Sidebar for the reference instance (autogenerated)         [COMMITTED]
migrate-docs.js                # Converts an extracted release into the reference instance  [COMMITTED]
scripts/sync-reference.mjs     # Downloads tarballs + generates the reference docs          [COMMITTED]
scripts/generate-llms.mjs      # Generates llms.txt / llms-full.txt                         [COMMITTED]
docusaurus.config.ts           # Reads releases.json to build the version dropdown          [COMMITTED]
src/  static/                  # Components, CSS, static assets                             [COMMITTED]

# ── generated at build time — all git-ignored ──
.releases-cache/               # Downloaded tarballs (cached by sha256)
releases/<version>/            # Extracted tarball for each published version
reference/                     # Reference `current` version            → served at /reference
reference_versioned_docs/      # Frozen older reference versions        → /reference/<version>
reference_versioned_sidebars/  # Autogenerated sidebar stub per version
reference_versions.json        # List of frozen reference versions
static/llms.txt, llms-full.txt # AI-discoverability files
```

## Local development

```bash
npm install
npm start          # runs `generate` then the dev server at http://localhost:3000
npm run build      # runs `generate` then production build → build/
npm run generate   # just (re)download + regenerate reference + llms files
npm run serve      # serve the production build locally
```

`prestart`/`prebuild` automatically run `npm run generate`, which:

1. fetches the upstream manifest (`releases.json` → `manifest` URL),
2. for each version in `releases.json` `versions`, resolves its docs tarball
   (`url` + `sha256`), downloads + verifies it (cached in `.releases-cache/`),
3. extracts to `releases/<version>/` and runs `migrate-docs.js` to produce
   `reference/` (current) and `reference_versioned_docs/version-<v>/`,
4. writes `reference_versioned_sidebars/*` and `reference_versions.json`,
5. generates `static/llms.txt` and `static/llms-full.txt`.

## Publishing a new reference version

The reference docs come from the upstream CI manifest:
`https://pantavisor-ci.s3.amazonaws.com/meta-pantavisor/releases.json` (each
version's `docs` entry carries the tarball `url` + `sha256`).

To publish a version, edit **`releases.json`** only:

```jsonc
{
  "manifest": "https://pantavisor-ci.s3.amazonaws.com/meta-pantavisor/releases.json",
  "current": "028-rc12",            // the default version shown
  "versions": ["028-rc12", "028-rc11", "028-rc10"]   // newest first
}
```

Commit `releases.json`; the build resolves each version's tarball (URL +
`sha256`) from the upstream manifest, downloads + verifies it, and regenerates
everything. The CI ingest workflow (`.github/workflows/docs-ingest.yml`)
automates this: given a `version` (or a `filename` to parse it from), it adds the
version to `releases.json` `versions`, sets it as `current`, verifies the build,
and opens a PR that changes only `releases.json`. **No generated docs are ever
committed.**

## How versioning works

Versioning applies to the **reference** instance only (the curated instance is
versionless):

| What Docusaurus calls it | This repo (generated) | URL path |
|---|---|---|
| `current` (default) | `reference/` | `/reference/` |
| Frozen version `028-rc10` | `reference_versioned_docs/version-028-rc10/` | `/reference/028-rc10/...` |

The set of versions and which one is `current` come from `releases.json`;
`docusaurus.config.ts` reads it to build the `docsVersionDropdown`
(`docsPluginId: 'reference'`).

## Broken links / warnings

Some generated reference documents contain cross-repository links (e.g.
`../../pantavisor-src/docs/...`) that cannot be resolved within this site. These
are emitted as **warnings**, not errors. They are a known limitation of the
source material and are a planned cleanup item before the canonical launch.
